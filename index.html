What is REST?
  • If you know HTTP, you know REST
  • Network architecture style, not a standard or protocol
  • Data/resource-oriented
  • Representational State Transfer, Roy Fielding [1]
    • "Representational State Transfer is intended to evoke an image of how a well-designed Web application behaves: a network of web pages (a virtual state-machine), where the user progresses through an application by selecting links (state transitions), resulting in the next page (representing the next state of the application) being transferred to the user and rendered for their use."
    • resource: referent of an URI  (wiki.../Resource_(Web))
    • representation: resource image (media-type)
    • state: the representation brings the client in a state
    • transfer: any state change following a link (a page is a graph of links)
    • Examples →    del.icio.us (*), twitter api
[1]: http://www.ics.uci.edu/~fielding/pubs/dissertation/top.htm, apache, http/1.1 specs
What is REST?
  • RESTful Constraints properties:
    1. Client–Server  independance, scalability
    2. Stateless:  any session state is held in the client, no client context stored on the server between requests reliability
    3. Cacheable:  clients can cache responses scalability, performance
    4. Layered system:  intermediary servers (load balancing, shared caches, firewalls, proxies, gateways) performance, security, compatibilities
    5. Code on demand (optional):  client-side scripts (js) or Java applets
    6. Uniform interface:  methods: GET, POST, PUT, DELETE or ...
  • RESTless:
    • Tunneling everything through GET, POST (params,...), Misusing cookies, Method-oriented protocols (RPC, SOAP*...)

CoRE Link Format
  • Aims:
    • media type: application/link-format
    • Web Linking for constrained devices, resource description
  • Use:
    • is requested addressing '
      /.well-known/core
      ' URI suffix
  • Ex requests:
    1. GET /.well-known/core
    2. GET /.well-known/core?rt=LightLux
    3. GET /.well-known/core?anchor=/sensors/temp

Observing
  • Aims:
    • Asynchronous subscription to a resource
    • Avoid continuous polling
  • Use:
    • observable resources have an "obs" flag in their core-link format
    • client subscribe with a GET & observe option
    • push notifications from server at any change
    • Type C/E Name Format Length Default
      10 Elective Observe uint 0-2 B (none)   
    • Obs relationship is ended with a GET from the client

CoAP

In brief

  • RESTful application-layer protocol over UDP
    • ex: coap://vs0.inf.ethz.ch/ (requires a coap browser extension (Copper, ...))
  • Simplified parsing, low overheaded, compared to RESTful HTTP interfaces
  • HTTP-CoAP mapping, android example (http://code.google.com/p/nodes)

Features

  • ↓ HTTP
    • Extensible header options but in ICMP-style: Type-Length-Value format
    • Resource abstraction, URI, media-type
    • Caching (Max-Age or ETag)*, proxy capabilities (Proxy-Uri)
    • Secured version: 
      "coaps://"
      using DTLS

Unlike HTTP, the cacheability of CoAP responses does not depend on the request method, but the Response Code (draft-ietf-core-coap-07#section-5.6)
CoAP
  • other different features
    • It's all binary
    • UDP based
      • asynchronous coms
      • unicast, multicast (≠ not specified yet)
    • Reliability
      • Transaction messages: (CON=Confirmable, NON=Non-Confirmable, ACK=Acknowledgment, RST=Reset)
      • exponential timeouts untill ack or max tries
    • piggy-backing, for CON responses, the ACK comes with the payload (if resource immediately available)
CoAP
  • others features
    • Token option to match responses to requests
    • Separate responses
